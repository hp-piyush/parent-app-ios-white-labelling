White-label rollout — hour budget (iOS)

iOS + bundled React Native — specified engineering hours per task (implement + test + fix). Estimates are for one additional brand after a shared codebase exists; see assumptions below. Figures reflect shipping distinct App Store apps (separate bundle IDs) from a single repository.

Why these phase weights. Each customer-facing app needs its own target, bundle identifier, provisioning, and App Store Connect record. The budget allows for code signing and CI export, APNs and entitlements, Universal Links and AASA (correct host, paths, and TEAMID.bundleid), App Store Connect work (privacy questionnaire, required screenshots, TestFlight), and React Native plus CocoaPods across schemes.

Totals. Sequential first brand: 118 h (sum of phase totals below). Phases can run in parallel;additional brand after the pipeline exists: 20 h (assets, Firebase iOS client, universal links / OAuth + AASA, App Store Connect + TestFlight slice, smoke test).

Assumptions. iOS UI work covers app-wide appearance tokens + splash, login, main shell, and settings — not a whole-repo purge of every hardcoded color. Runtime branding assumes the API contract and sample payloads exist before that block starts. One Apple Developer team; each brand still needs its own bundle ID, App ID, and distribution provisioning profile (or automatic signing with correct team + bundle).

What the backend must provide (no implementation timeline)

The mobile client depends on the contracts below for runtime behavior. Bundle identifiers and the apple-app-site-association file must stay aligned with what ships in Xcode and on your HTTPS hosts.

Need What to supply (concrete) Why / how the app uses it
Tenant / org resolution Documented rule: tenant/school id (header, login response, deep link, install config). Staging users for multiple tenants. Runtime branding and cache keys depend on a stable rule.
Branding payload Versioned JSON (or OpenAPI): colors, logo URLs, optional strings; staging examples; optional ETag/version. The 22 h runtime block maps fields into UI; vague contracts waste time.
Legal & support URLs Per-tenant or per-app privacy, terms, support URLs (or agreement they live only in ASC/plist). Store and in-app must stay consistent per listing.
Push semantics Topic/segment naming, subscribe/unsubscribe rules, test sends per tenant; or “server-only, client unchanged beyond plist.” Avoids cross-tenant push mistakes.
Deep link & OAuth alignment HTTPS hosts/paths per env; redirect URIs per iOS bundle ID; sample magic links; AASA paths and TEAMID.bundleid for each shipped app. Universal links and OAuth fail when host, path, or appID do not match.

Anything not in this payload remains the client’s responsibility: bundle identifier, App Icon, default local theme, Firebase → bundle ID mapping, code signing, and App Store Connect metadata ownership.
